AWS IAM & IAM Identity Center

1) Scope & audience

  • Audience: Cloud/DevOps/Security engineers and admins
  • Scope: Human access to AWS accounts (console + CLI) using:
    • IAM Identity Center for workforce SSO across multiple AWS accounts (recommended)
    • IAM (direct) for service roles, break-glass, and limited cases

2) Key concepts (simple mapping)

  • Identity Provider (IdP): Where users authenticate (password, MFA, conditional access).
  • IAM Identity Center: Central place to assign users/groups to AWS accounts with permission sets.
  • Permission set: Template of permissions (managed policies + optional inline policy + session duration). When assigned, it creates an IAM role in the target account.
  • IAM role: The actual identity used inside each AWS account to access services.
  • IAM (direct): Best for workloads (EC2/Lambda/ECS) and cross-account access via roles. Not best for humans.

3) Step-by-step: Implement IAM Identity Center (recommended path)

Step 1 — Set up AWS Organizations (multi-account baseline)

  1. In the Management account, open AWS Organizations.
  1. Create/verify your Organization.
  1. Create accounts (minimum):
    • Dev
    • Test
    • Prod
  1. (Recommended) Add security foundation accounts:
    • Log Archive
    • Security Tooling

Output: You have separate AWS accounts for isolation and better access control.

Step 2 — Enable IAM Identity Center

  1. Open IAM Identity Center in the chosen Region.
  1. Select Enable.
  1. Confirm it is integrated with your Organization.

Output: Identity Center becomes the central control plane for workforce access.

Step 3 — Configure identity source (IdP integration)

Choose one:

Option A: Built-in directory (fastest for labs/small teams)

  1. IAM Identity Center → Settings → Identity source
  1. Keep Identity Center directory
  1. Create users and groups in Identity Center

Option B: External IdP (recommended for enterprises)

  1. IAM Identity Center → Settings → Identity source
  1. Choose external IdP method (SAML/OIDC as supported)
  1. Exchange metadata/config between IdP and AWS
  1. Validate login end-to-end

Output: Users authenticate through your chosen identity system.

Step 4 — Create users and groups

Create groups aligned to job functions (example):

  • AWS-Admins
  • AWS-Developers
  • AWS-ReadOnly
  • AWS-Billing (optional)

Output: Group-based access (easier to manage than per-user permissions).

Step 5 — Design permission sets (role-based access model)

Create permission sets that match responsibilities.

Recommended starter set

  • Admin: AdministratorAccess
  • PowerUser: PowerUserAccess + additional restrictions if needed
  • ReadOnly: ReadOnlyAccess
  • Billing: Billing/finance appropriate policies (avoid wide service access)

Guardrails

  • Keep permission sets small and reusable
  • Prefer AWS managed policies as a baseline
  • Add inline policy only when necessary
  • Set session duration thoughtfully (e.g., 1–8 hours)

Output: Standardized permission “packages” you can assign consistently.

Step 6 — Assign access to accounts (who gets what, where)

  1. IAM Identity Center → AWS accounts
  1. Select an account (Dev/Test/Prod)
  1. Click Assign users or groups
  1. Choose the Group + Permission set

Example assignment model

  • AWS-Admins → Admin → Dev/Test/Prod
  • AWS-Developers → PowerUser → Dev only
  • AWS-ReadOnly → ReadOnly → Prod (and optionally all)

Output: Users can see accounts in the AWS access portal.

Step 7 — Understand the role that gets created in each account

When you assign a permission set, AWS creates roles like:

  • AWSReservedSSO_<PermissionSetName>_<random>

These roles:

  • Are assumed via the Identity Center login flow
  • Should generally not be edited directly (manage via permission sets)

Output: Each account now has IAM roles tied to Identity Center assignments.

Step 8 — User login flow (console + CLI)

Console

  1. User opens the AWS access portal URL
  1. Authenticates via IdP
  1. Picks account + role
  1. Accesses AWS console

CLI (recommended)

  • Use AWS CLI v2 with SSO profiles
  • Configure per-account profiles; users authenticate once and get temporary credentials

Output: No long-term access keys needed for humans.

4) Direct IAM (traditional) — when to use and how (use sparingly)

When direct IAM is acceptable

  • Workload identities: EC2/Lambda/ECS assume roles (best practice)
  • Break-glass admin: tightly controlled emergency access
  • Third-party systems that cannot use Identity Center (evaluate carefully)

Avoid for human users

  • Long-term access keys
  • Local IAM users per account
  • Manual permission drift

Minimal steps (only if required)

  1. Create an IAM role (preferred) or IAM user (last resort)
  1. Attach least-privilege policy
  1. Enforce MFA (for console users)
  1. Rotate/disable access keys regularly if used

5) Best practices (production)

Security

  • Use least privilege (start narrow, expand as needed)
  • Enforce MFA (IdP MFA + root MFA)
  • Disable or remove long-term access keys for humans
  • Use permission boundaries and SCPs where appropriate

Governance

  • Use AWS Organizations SCPs to set account-level guardrails
  • Standardize naming for groups and permission sets
  • Document access patterns and approval workflow

Observability & audit

  • Enable CloudTrail in all accounts (centralized)
  • Use AWS Config for drift detection
  • Review Identity Center assignments periodically (quarterly access reviews)

6) Glossary (quick)

  • AuthN: Authentication (who you are)
  • AuthZ: Authorization (what you can do)
  • SSO: Single Sign-On
  • SCP: Service Control Policy (Organization-level guardrails)
